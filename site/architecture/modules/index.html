    üì¶ Module Details¶

    Layer 0: NomadCore (Foundation)¶

    Purpose: Provides fundamental utilities and data structures used throughout NOMAD.

    Status: ‚úÖ Complete

    Dependencies: None (zero external dependencies)

    Key Components: - Math library (vectors, matrices, interpolation) - Threading primitives (lock-free queues, thread pools) - File I/O system (cross-platform file operations) - Logging system (structured logging) - Memory management (custom allocators) - String utilities (UTF-8, formatting)

    Public API: 

    #include "NomadCore/Math/Vector.h"
#include "NomadCore/Threading/ThreadPool.h"
#include "NomadCore/IO/File.h"
#include "NomadCore/Logging/Logger.h"

    View NomadCore Details ‚Üí

    Layer 1: NomadPlat (Platform Abstraction)¶

    Purpose: Provides unified API for platform-specific operations.

    Status: ‚úÖ Complete (Windows), üöß In Progress (Linux), üìÖ Planned (macOS)

    Dependencies: NomadCore

    Key Components: - Window management (create, resize, event handling) - Input system (keyboard, mouse, touch) - File dialogs (open/save) - System information (CPU, memory, OS) - High-resolution timers

    Supported Platforms: | Platform | Status | Implementation | |----------|--------|----------------| | Windows 10/11 | ‚úÖ Complete | Win32 API | | Linux | üöß In Progress | X11, Wayland planned | | macOS | üìÖ Planned Q2 2025 | Cocoa |

    Public API: 

    #include "NomadPlat/Window.h"
#include "NomadPlat/Input.h"
#include "NomadPlat/FileDialog.h"

    Layer 2: Framework Modules¶

    NomadUI (GPU-Accelerated Renderer)¶

    Purpose: Custom OpenGL-based UI framework with modern features.

    Status: ‚úÖ Complete

    Dependencies: NomadCore, NomadPlat

    Key Components: - OpenGL 3.3+ renderer - NanoVG vector graphics - Layout engine (flexbox-inspired) - Widget library (buttons, sliders, inputs) - Theme system (dark/light modes) - SVG icon system - Animation system

    Performance: - Adaptive FPS: 24-60 FPS - MSAA anti-aliasing - Texture atlasing - Dirty rectangle optimization

    Public API: 

    #include "NomadUI/Renderer.h"
#include "NomadUI/Widget.h"
#include "NomadUI/Layout.h"
#include "NomadUI/Theme.h"

    View NomadUI Details ‚Üí

    NomadAudio (Audio Engine)¶

    Purpose: Professional audio processing with ultra-low latency.

    Status: ‚úÖ Complete

    Dependencies: NomadCore, NomadPlat

    Key Components: - WASAPI integration (Windows) - RtAudio backend (cross-platform) - 64-bit audio pipeline - Multi-threaded DSP - Sample-accurate timing - Low-latency design (<10ms)

    Audio Capabilities: - Sample rates: 44.1kHz - 192kHz - Buffer sizes: 64 - 4096 samples - Bit depths: 16, 24, 32-bit integer and float - Channels: Up to 32

    Public API: 

    #include "NomadAudio/AudioEngine.h"
#include "NomadAudio/AudioDevice.h"
#include "NomadAudio/AudioBuffer.h"

    View NomadAudio Details ‚Üí

    NomadSDK (Plugin System)¶

    Purpose: Plugin hosting and extension system.

    Status: üìÖ Planned Q2 2025

    Dependencies: NomadCore, NomadPlat

    Planned Features: - VST3 plugin hosting - Effect processing (inserts, sends) - Parameter automation - MIDI routing - Plugin scanning and caching

    Planned API: 

    #include "NomadSDK/PluginHost.h"
#include "NomadSDK/Effect.h"
#include "NomadSDK/Automation.h"

    Status: üöß Active Development

    Inter-Module Communication Patterns¶

    sequenceDiagram

    App->>Audio: Initialize Engine

    Plat-->>Audio: Device List

    loop Main Loop
        Plat->>UI: Input Events
        UI->>App: UI Events
        App->>Audio: Transport Control

    end

    Data Flow Patterns¶

    1. Event-Driven (UI ‚Üí Application)¶

    // UI generates events
ui.onButtonClick([&](Button* btn) {
    app.handlePlayButton();
});

    2. Callback-Based (Audio ‚Üí Application)¶

    // Audio engine calls back on audio thread
audio.setCallback([&](AudioBuffer& buffer) {
    app.processAudioBlock(buffer);
});

    3. Polling (Application ‚Üí Platform)¶

    // Application polls platform state
while (app.running) {
    plat.pollEvents();
    app.update();
}

    üéØ Design Guidelines¶

    Module Design Principles¶

    1. Single Responsibility
    2. Each module has one clear purpose

    3. Avoids feature creep and bloat

    4. Minimal Dependencies

    5. Only depend on lower layers

    6. Minimize coupling between modules

    7. Interface Segregation

    8. Small, focused interfaces

    9. Clients only depend on what they use

    10. Dependency Inversion

    11. Depend on abstractions, not implementations

    12. Use interfaces and abstract classes

    13. Open/Closed Principle

    14. Open for extension
    15. Closed for modification

    Adding New Modules¶

    When adding a new module to NOMAD:

    1. Determine Layer
    2. Identify which layer the module belongs to

    3. Ensure it only depends on lower layers

    4. Define Public API

    5. Create clear header-only interface

    6. Document all public functions

    7. Write Tests

    8. Unit tests for core functionality

    9. Integration tests with dependent modules

    10. Update Documentation

    11. Add to this hierarchy page

    12. Create detailed module documentation

    13. Update CMake

    14. Add to build system
    15. Configure dependencies
